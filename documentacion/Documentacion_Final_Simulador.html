1. Descripción del Proyecto

SimuladorMiniPC es una aplicación de escritorio educativa que implementa un simulador de sistema operativo de tiempo discreto. Modela los mecanismos fundamentales de gestión de procesos: ciclo de vida, planificación de CPU, asignación de memoria, interrupciones y dispositivos de E/S.

Concepto simuladoMecanismo
MultiprogramaciónHasta 5 procesos concurrentes en memoria
Estados de proceso7 estados: NEW → READY → RUNNING → BLOCKED → TERMINATED
PlanificaciónPolítica activa actual: FCFS (infraestructura extensible para otras políticas)
Gestión de memoriaRAM 700 celdas, first-fit, área del SO (celdas 0–19)
Memoria virtualTraducción base+offset por proceso; validación de offsets
InterrupcionesINT 20H, INT 10H, INT 09H, INT 21H
E/S con latenciasTeclado (MAX_VALUE), Pantalla (1 tick), Disco (5 ticks)
EstadísticasUtilización CPU, throughput, tiempos de espera/retorno/respuesta

Stack Tecnológico

ComponenteVersiónRol
Java25Lenguaje principal
JavaFX24Scene Graph, ObservableList, bindings, Timeline, TableView
NetBeans Ant—build.xml, nbproject/ — compilación y empaquetado
JSON (sin libs)—MemoryConfig, DiskConfig y AssemblerConfig parsean con regex propio
PlantUML—Diagramas UML en documentacion/diagramas/

2. Inventario de Paquetes

Esta sección presenta el mapa modular del simulador desde una perspectiva de arquitectura. Su propósito es explicar cómo se distribuyen responsabilidades por paquete para reforzar mantenibilidad, trazabilidad de cambios y estrategia de pruebas.

Total: 14 paquetes, 43 clases Java.

simuladorminipc (raíz)
├── assembler    ← parser léxico/sintáctico del ASM propio
├── clock        ← reloj global monotónico
├── cpu          ← CPU virtual + Dispatcher
├── fx           ← capa de presentación JavaFX
├── interrupt    ← cola FIFO de interrupciones
├── io           ← dispositivos de E/S con latencias
├── kernel       ← núcleo orquestador (Fachada)
├── memory       ← RAM, first-fit, traducción virtual, area del SO
├── model        ← PCB, RegisterSet, ProcessState, IORequest
├── process      ← admisión, colas y transiciones de estado
├── scheduler    ← política activa FCFS (otras políticas en código)
├── stats        ← métricas de rendimiento
└── storage      ← disco secundario simulado

Análisis léxico y sintáctico. Transforma archivos .asm en listas de objetos Instruction listos para ejecutar.

RegisterSetac, ax, bx, cx, dx, ir, pc, ah, al, dxString, zeroFlag, stack[5], stackPointer.

simuladorminipc.scheduler

Interfaz SchedulingAlgorithm con selectNextProcess(List<PCB>, PCB currentRunning) → PCB y getName(). En la operación actual del simulador, la política activa es FCFS.

ClaseCriterioApropiativo
FCFSSchedulerMenor arrivalTime (desempate: menor PID)No

SchedulingPolicyManager conserva la infraestructura para otras políticas, pero la ejecución vigente se documenta con FCFS.

ClaseResponsabilidad
StatisticsManagertotalTicks, activeTicks, finishedProcesses. Calcula utilización, throughput, tiempos promedio.
ProcessStatsSnapshot inmutable al terminar: PID, tiempos de inicio/fin/espera/retorno/respuesta/servicio, ratio.

simuladorminipc.storage

ClaseResponsabilidad
DiskString[1080]. INDEX_SIZE=10. createFile/openFile/writeFile/readFile/deleteFile/storeProgram.

3.3 Especificación de Clases

La siguiente sección documenta objetivo, atributos, constructores y métodos clave para facilitar mantenimiento y trazabilidad.

Clase: PCB (simuladorminipc.model)

/**
 * Objetivo:
 *   Representar el Bloque de Control de Proceso con toda la información
 *   necesaria para planificación, ejecución, memoria y estadísticas.
 *
 * Atributos principales:
 *   @field pid               Identificador único del proceso.
 *   @field state             Estado actual del ciclo de vida del proceso.
 *   @field instructions      Lista inmutable/lógica de instrucciones ASM.
 *   @field registers         Snapshot de registros de CPU del proceso.
 *   @field memoryBase/limit  Rango físico asignado en RAM.
 *
 * Métodos clave:
 *   @method getCurrentInstruction()
 *     @return Instruction instrucción actual según PC.
 *   @method hasMoreInstructions()
 *     @return boolean true si el proceso puede seguir ejecutando.
 */

Clase: Kernel (simuladorminipc.kernel)

/**
 * Objetivo:
 *   Orquestar todos los subsistemas del simulador y ejecutar el pipeline
 *   de 10 etapas por tick.
 *
 * Atributos principales:
 *   @field cpu, dispatcher, policyManager   Núcleo de ejecución y planificación.
 *   @field queueManager, stateManager       Gestión de colas y transiciones.
 *   @field memoryManager, vmManager, disk   Gestión de memoria y almacenamiento.
 *   @field ioManager, interruptManager      Interrupciones y E/S.
 *
 * Constructores:
 *   @constructor Kernel(int ramSize, int diskSize)
 *     @param ramSize  tamaño de RAM simulada en celdas.
 *     @param diskSize tamaño de disco simulado en celdas.
 *
 * Métodos clave:
 *   @method boot() Inicializa reloj, banderas y estado del kernel.
 *   @method executeTick() Ejecuta un ciclo completo del SO simulado.
 *   @method loadProgram(File file, int arrivalTime, int priority)
 *     @param file archivo asm a ensamblar y cargar.
 *     @param arrivalTime tick de llegada del proceso.
 *     @param priority prioridad inicial (numérica).
 */

Clase: CPU (simuladorminipc.cpu)

/**
 * Objetivo:
 *   Ejecutar instrucciones del proceso activo y producir eventos de ciclo.

 * Atributos principales:

/**

4. Pipeline de Ejecución

Esta sección explica, de forma secuencial, cómo se ejecuta el sistema operativo simulado en tiempo discreto. Cada tick mantiene un orden estable para garantizar reproducibilidad, depuración y consistencia métrica.

4.1 Ciclo Tick — 10 Etapas

① clock.tick()
② processManager.loadArrivingProcesses()
③ ioManager.tick()
④ scheduleAndDispatch()
⑤ cpu.executeCycle()
⑥ handleCycleEvent()
⑦ quantum check (RR)
⑧ statisticsManager.onTick()
⑨ fireTickCompleted() → UI
⑩ allFinished() check

El método executeTick() siempre sigue este orden fijo. La idea es: primero se actualiza el tiempo y el estado interno del sistema, luego se ejecuta CPU, y al final se publican resultados y se verifica el cierre de la simulación.

EtapaQué haceResultado inmediato
① clock.tick() Incrementa el reloj global del sistema. El tick actual avanza en 1; todas las decisiones del ciclo usan este nuevo valor.
② processManager.loadArrivingProcesses() Revisa procesos en NEW cuyo tiempo de llegada ya se cumplió e intenta admitirlos. Si hay RAM disponible pasan a READY; si no, permanecen en NEW.
③ ioManager.tick() Avanza los contadores de latencia de dispositivos de E/S. Los procesos con E/S finalizada se desbloquean y regresan a READY.
④ scheduleAndDispatch() El planificador elige el siguiente proceso elegible y el dispatcher realiza el cambio de contexto. La CPU queda cargada con el proceso que correrá en este tick (si existe alguno).
⑤ cpu.executeCycle() Ejecuta una unidad de trabajo de CPU sobre el proceso actual. Devuelve un CycleEvent que resume lo ocurrido en la ejecución.
⑥ handleCycleEvent() Interpreta el evento de CPU y aplica transiciones de estado o interrupciones según corresponda. Se procesa el caso: normal, finalización, error, solicitud de E/S, teclado o salida a pantalla.
⑦ Quantum check (extensión futura) Reservado para políticas con quantum. En FCFS activo esta verificación no produce desalojo. Sin efecto operativo en FCFS; queda como punto de extensión del pipeline.
⑧ statisticsManager.onTick() Actualiza métricas globales del sistema y persiste el BCP en área del SO. Se actualizan totalTicks, activeTicks y snapshots de estado para análisis.
⑨ fireTickCompleted() Notifica a los listeners (UI/controlador) que el tick terminó. La interfaz se refresca con colas, CPU, memoria, disco y registros actualizados.
⑩ allFinished() check Comprueba si ya no quedan procesos por ejecutar. Si todo terminó, el kernel marca halted=true y dispara onAllFinished().

Resumen práctico: cada tick ejecuta exactamente una vuelta completa del sistema operativo simulado: admitir, desbloquear, planificar, ejecutar, registrar y notificar.

5. Gestión de Memoria

Esta sección detalla la asignación de memoria principal, la validación del direccionamiento virtual y los parámetros configurables para ejecutar escenarios con diferentes perfiles de carga.

5.1 Algoritmo First-Fit

Estructura de RAM (700 celdas):
  Celdas 0–19   → Área del SO (BCPs de procesos activos)
  Celdas 20–699 → Área de usuario (instrucciones de procesos)

Al admitir proceso P con memoryRequired = N:
  Para cada MemoryBlock b (ascendente por dirección):
    si b.free && b.size >= N:
      b.free = false;  b.pid = P.pid
      P.memoryBase = b.base;  P.memoryLimit = b.base + b.size - 1
      Escribe instrucciones en RAM[b.base .. b.base+N-1]
      return true
  return false  ← proceso permanece en NEW

5.2 Traducción de Memoria Virtual

Dirección física = P.memoryBase + virtualOffset

Validación:
  si virtualOffset < 0 || virtualOffset >= (P.memoryLimit - P.memoryBase):
    → CycleEvent.error("Acceso fuera de rango de memoria")
ParámetroValor actualConfigurable en
ramSize700memory-config.json
virtualMemorySize80disk-config.json
diskSize1080disk-config.json
maxInstructions80assembler-config.json

11. Manual de Usuario

Este manual describe la operación funcional del simulador desde la perspectiva del usuario final, con una guía progresiva: inicio, flujo recomendado, lectura de paneles y resolución de incidencias.

11.1 Inicio rápido

  1. Abra el proyecto simuladorMiniPC en NetBeans.
  2. Ejecute F6 para iniciar la aplicación.
  3. Pulse Cargar archivos y seleccione programas .asm.
  4. Use Paso para ejecución detallada o Auto-todos para ejecución continua.

11.2 Flujo operativo recomendado

  • Cargar 2-3 programas ASM válidos para observar interacción entre colas.
  • Verificar transición de NEW a READY tras admisión en memoria.
  • Revisar eventos de pantalla/teclado/disco en bitácora.
  • Al terminar, usar panel de métricas para analizar rendimiento.

11.3 Guía de interpretación de paneles

PanelQué observarCuándo usarlo

11.4 Manejo de incidencias frecuentes

  • Archivo ASM rechazado: revisar línea reportada y sintaxis de operandos.
  • Proceso no avanza: verificar si está BLOCKED por teclado o disco.
  • Entrada de teclado pendiente: proveer valor cuando aparezca solicitud INT 09H.
  • Resultados inesperados: reiniciar simulación y repetir en modo paso a paso.
